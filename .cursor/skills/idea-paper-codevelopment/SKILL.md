# 从想法到 idea/paper.md 体例

## 目标

与用户协同把模糊想法推进到**可落笔**状态，并最终生成与仓库中 `idea/paper.md` **同构**的中文 Markdown 论文草稿（含摘要、关键词、分节正文、公式与参考文献）。

## 领域泛化

本技能**不限定**数据库、排序或 LLM：适用于工程系统、风险监测、安全合规、金融风控、公共卫生等。Agent 须在首轮或用户说明领域后，**把追问与参考文献锚定到该领域的术语、监管口径、评价惯例**（例如风险监测常问：监测对象与事件定义、领先/滞后指标、告警成本与误报代价、数据可得性与标签、与监管或行业框架的对齐），避免机械套用 ML 论文话术。

## 核心行为

1. **以想法为锚提问**：每次回复优先列出**少量高价值追问**（建议 3–7 条），针对当前信息缺口；避免一次性冗长问卷。
2. **补充参考材料**：根据已述主题，**主动给出**可检索的文献方向（经典论文、综述、近年的代表工作），并说明每条与用户问题的关联；若环境允许网络检索，应检索后给出**可核实**的题名、年份、venue 或作者，供用户核对。
3. **评价与建议**：用简短小节给出：**亮点 / 风险或漏洞 / 可加强的论证 / 实验与对比建议**；语气建设性、具体可改。
4. **迭代终止条件**：当下方「就绪度检查」全部可填且用户确认后，再输出完整 `idea/paper.md` 格式文稿；否则继续追问与补材料。

## 就绪度检查（通过后再生成全文）

在生成终稿前，在对话中显式勾选或口头确认：

- [ ] **问题**：任务定义、输入输出、规模或场景清晰；与现有做法的差异一句话能说清。
- [ ] **贡献**：2–4 条可检验的贡献点（方法/理论/效率/可解释性等），无空泛形容词堆砌。
- [ ] **方法**：主要符号、核心算法或流程可概述；关键假设已声明。
- [ ] **相关工作**：至少 4–6 个方向上有「谁做了什么、与本文关系」；缺口已识别。
- [ ] **实验（或等价验证）**：数据/设置、对比基线、主指标、主要结果趋势；形式可以是实验表、**回溯/仿真、案例研究、专家评估或问卷**，但须说清**评价对象与可重复性**（可先用占位表或占位小节标题）。
- [ ] **摘要要素**：背景一句、痛点、方法要点、定量或定性结论各有一句。

未满足时：**不要**输出长篇终稿；继续针对未勾选项提问并补参考。

## 追问题库（按需选用，勿一次全抛）

**通用（多数领域）**

- 要解决的核心**决策、预测或流程改进**是什么？**干系人**是谁（谁用、谁担责）？
- **输入与输出**、时间粒度、是否在线/离线？
- 与**现行做法或标准基线**相比，多出来的**机制或证据**是什么？
- **主指标**与**基线**各是什么？若没有真实数据，计划用什么**公开数据、仿真、历史回放或专家评估**？
- 有哪些**必须引用的经典文献**与**近两年的对标工作**（含行业报告、标准、白皮书，若该领域以此为主）？

**计算机/学习系统类（可选用）**

- **复杂度或规模**瓶颈在哪里？如何论证可扩展？
- **失败模式**（如幻觉、漂移、不一致、冷启动等）与应对？

**风险监测 / 风控 / 安全运营（可选用）**

- **风险事件或损失**的操作化定义？监测是事前、事中还是事后？
- **误报/漏报**的业务代价与可接受权衡？是否有监管或审计要求？
- **数据源**可信度、延迟、对抗与数据缺失时怎么办？
- 是否需要**与框架对齐**（如 NIST、ISO、行业监管条款）并说明映射关系？
