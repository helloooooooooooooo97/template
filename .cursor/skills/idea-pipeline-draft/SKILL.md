# 想法 → 待验证草稿

## 目标

把模糊想法压成一份**待验证草稿**：写清「要证什么、怎么证、证不过怎么办」。**不写**论文全文体例、不写终稿级综述或实验叙事。

草稿**必须**包含三块：

1. **可在网上核实**的参考资料（URL、标准号+机构、或可执行的检索式）。
2. **数据处理流水线**：端到端环节，**每一环节至少两种可行路径**（主路径 + 备选），并简短对比适用条件、复杂度、风险。
3. **分阶段假设**：每环节说明**为何能进入流水线**（输入可用性、质量/分布前提、与上下游接口、可推广性等）；假设与**验证思路**成对出现，并区分**工作假设**（待验证）与**已成立前提**（有依据）。

## 工作流

1. **锚定问题**：每次回复优先 **3–7 条**高价值追问，补齐领域、数据形态、目标产出、约束（合规/延迟/成本）、评价方式；避免一次冗长问卷。
2. **流水线建模**：按领域裁剪阶段（如采集 → 清洗 → 特征/表示 → 模型或规则 → 服务或报告）；每环节 **2+ 路径**。
3. **分阶段假设**：每环节写清进入理由与依赖前提；说明**若假设错**，流水线表现或哪条路径失效。
4. **网上可查资料**：环境允许时用 **Web 检索**；每条须可核验（官方文档 URL、标准编号+发布机构、arXiv/DOI/会议页等）；**禁止编造链接**；仅能提示检索时写清 **关键词 + 建议来源类型**。
5. **迭代**：根据用户回复更新流水线、假设、参考与验证项；直到用户表示「这版可拿去验证」或追问收敛。用语突出**验证清单与下一步**，避免「已定稿」「论文终稿」等暗示。

## 输出格式

- 完整结构见 [output-template.md](output-template.md)。
- 在对话中可先给缩略版，再按需展开；定稿时对齐模板中的分环节表、参考资料分组、**验证计划总表**、待确认清单。

## 质量约束

- 全文术语一致；假设**必须可检验**，并写清推翻后的后果或回退。
- 参考资料仅用于支撑某路径或某假设，**够用即可**，不扩写成长篇综述。
