# 发明专利交底书（Markdown + PNG 图示）

## Instructions
你需要完成一个端到端流程：
1. 与用户交互深化：把用户给的 `idea` 从“概念/想法”澄清到“可写进交底书第 1-3 部分”的颗粒度。
2. Agent 拆解与提示词包生成：把澄清后的要点整理为 Prompt Pack（供通用大模型生成文字与图示）。
3. 生成图示与文案：先形成图逻辑（节点/连线语义 + 图注证明点），再直接生成独立 PNG 图片，并输出第 1-4 部分的最终 Markdown（最终文档不输出 Mermaid 代码块）。

### 0) 触发与输入检查
- 若用户未提供 `idea` 或只说了很模糊的方向：只追问最多 3 个问题让 `idea` 变得可加工。
- 若用户已给出一定细节：进入“阶段 1”按栏目推进澄清。

### 1) 阶段 1：信息骨架（先问后写）
你要先输出一个“技术骨架草案”（要点级，不写长文），并让用户确认方向：
1. 技术对象/应用场景（软件/系统/方法/装置？）
2. 预期解决的问题（对应“现有技术缺陷”的方向）
3. 初步模块/步骤设想（用户提供多少就写多少，不够就用占位）
4. 需要的图示类型（默认“流程图/结构图”，最终以 PNG 输出）

骨架草案输出后，追问用户确认并补齐：只问“会直接影响第 3 部分连接关系”的问题。

### 2) 阶段 2：对齐交底书模板栏目（多轮澄清）
你要按下面栏目逐步澄清，并确保第 3 部分能写到“具体改进点如何实现发明目的”：

#### 2.1 栏目 1：现有技术的现状及缺陷
向用户确认（每类最多问 2 个问题）：
1. 现有技术通常怎么做（流程/模块级描述）
2. 它的缺陷是什么（成本高/速度慢/可靠性差/效果差/使用不方便/副作用/难以扩展等，尽量对应你的方案）
3. 缺陷发生在流程的哪一步（帮助后面“改进点如何对上缺陷”）

你需要在内部形成“缺陷 -> 对应改进点来源”的对照表（不必展示给用户，但最终要能落到文案里）。

#### 2.2 栏目 2：整体构思
向用户确认：
1. 本发明的目的（用一句话描述：要解决哪个技术问题）
2. 你具体改进了哪一部分（模块/流程步骤/机制/约束/算法/连接关系/参数策略）
3. 改进后的关键技术效果（先给要点，后面会写入 Markdown）
4. 创新点提炼与确认（必须执行）：
   - 先基于用户 `idea` 提炼 2-4 条“明确创新点”（例如：流程创新、结构创新、约束/机制创新、工程实现创新）。
   - 每条创新点都要能回答三件事：相比现有技术新在哪里、为什么不是常规拼接、对应解决了什么问题。
   - 如果无法从当前 `idea` 中稳定提炼创新点（信息不足/表述模糊/与现有做法差异不清），必须向用户追问确认后再继续后续生成。
   - 在进入阶段 3（Prompt Pack）前，必须完成“创新点确认”，并在内部形成“创新点 -> 第3部分改进点段落”的映射关系。

#### 2.3 栏目 3：技术方案详细阐述（最关键）
向用户确认（缺任何一类都要继续追问）：
1. 功能模块/装置部件清单（如果是软件，至少到“功能模块”；如果是系统/方法，至少到“步骤/子模块”）
2. 模块之间的连接关系（数据流/控制流/触发条件/调用关系）
3. 关键改进点“如何实现”（把差异点落到可执行步骤或具体处理逻辑）
4. 替代方案（至少 1 个备选实施方式：不同的实现路径/不同的参数策略/不同的模块组合）

你需要输出一个“技术方案草案（可写第 3 部分）”，供用户确认颗粒度。

### 3) Agent 拆解与 Prompt Pack 生成（内部产物）
在用户确认完技术方案后，你要产出三份 Prompt Pack（只用于你生成文案与图示；不需要让用户逐字看，但可以按需总结给用户审查）：
1. Prompt Pack-1：栏目 1（现有技术与缺陷，缺陷与改进点要能对应）
   - 在生成栏目 1 前，必须先进行网络检索（专利/论文/标准/权威技术博客均可），提取至少 2-4 条代表性参考来源，并在生成文案时以行内引用或参考文献列表的形式给出来源依据。
   - 栏目 1 的写作以连续段落为主，先概述“现有技术如何做”，再自然过渡到“其缺陷是什么”，最后把缺陷与本发明改进点逐一在叙述中对齐（允许少量编号用于对应关系）。
2. Prompt Pack-2：栏目 2（整体构思：目的 + 改进点清单）
3. Prompt Pack-3：栏目 3（详细阐述 + 图示 PNG + 替代方案）

Prompt Pack-3 的输出要求：
- 必须生成至少 1 张“图逻辑结构”（用于驱动图片生成），必要时 2 张或更多（图 1 主流程、图 2 关键模块交互/数据流）。
- 图逻辑中节点与连线要覆盖“模块清单 + 连接关系 + 执行顺序”
- 图示解释信息：用于后续写入交底书细节：
  - 图注：不少于 3 条要点（每条对应图中一个关键链路/改进点）
  - 逐节点解释：按节点名逐条说明该节点的作用、输入/输出（至少覆盖主要节点）
  - 逐连线解释：对关键连线（至少覆盖 4 条）说明连线语义（数据/控制/触发条件）与为何能实现对应改进点
  - 对照映射：明确标注每张图对应第 3 部分的哪些改进点编号/段落要点（如：改进点 1/2/3，对应 3.x 的要点）
- 同时为每张图给出“PNG 生成与落盘规格”，并在后续步骤中**实际执行**独立 PNG 生成与落盘（不要只输出 prompt）：
  - 图名/图号（如：图 1、图 2）
  - 对应图逻辑结构的唯一标识（避免生成到错误的图）
  - 默认保存目录（若未指定则使用：`./patent_outputs/diagram_png/`）
  - 默认文件名规则（如：`{发明设计名称}-图1-{时间戳}.png`）
  - 默认相对路径引用方式（例如在 Markdown 中使用 `![图1](./patent_outputs/diagram_png/xxx.png)`）
 - 对于每张图，额外输出“PNG 生成执行状态”与“失败原因”（供回退策略使用）：
   - 生成状态：`success` 或 `fail`
   - 若 `fail`：输出一条简短原因（例如“图像生成能力不可用/落盘权限不足/生成模型失败”等），并保持与后续文档中的注释一致
- 同时为“独立图片生成”提供输入规格（不依赖任何 Mermaid 渲染器）：
  - Image generation prompt：基于图逻辑结构的节点/连线语义 + 图示解释，生成用于图像生成模型的指令（包含：布局、节点形状、箭头含义、字体大小/中英文规则、图注位置、整体风格）
  - Image generation layout constraints：至少包含节点排列策略（例如自上而下流程/分组对齐）、最小间距、避免文字过密

### 4) 最终输出：Markdown 专利交底书（与模板栏目对齐）
你最终只输出一个 Markdown 文档（不要生成 docx），并且要把完整 Markdown **保存成真实的 `.md` 文件**（落盘后可在编辑器里看到）：
1. 基础信息区
   - 发明设计名称：从用户 idea 推断，缺失则 `TODO(待填写)`
   - 设计发明人/所属单位/电话号码/Email/撰写日期：缺失则 `TODO(待填写)`
2. `## 1．现有技术的现状及缺陷`
   - 先触发网络搜索（若环境支持），提取至少 2-4 条相关现有技术参考来源；随后以连续段落描述现有技术的典型实现方式，再用连续叙述写出缺陷来源与表现；最后在同一段落或相邻段落中完成“缺陷 -> 对应本发明改进点”的对齐说明。
   - 在文末增加一个简短的“参考文献/来源（与本栏目相关）”小节，列出你在搜索中使用的关键来源（至少 2 条）。
3. `## 2．本发明的整体构思`
   - 明确目的 + 改进了哪一部分
4. `## 3．本发明技术方案的详细阐述`
   - 必须包含（建议按下面顺序组织，允许你在 `## 3` 内增加更细二级/三级标题）：
     - 写作风格与内容丰满度要求：
      - **内容密度强制要求：** 每个核心模块（Section 3.1 等）必须由 2-3 个连续的长段落组成。严禁仅用一段话概括一个模块。
      - **“多维叙述”链路：** 每一段必须遵循“输入定义 -> 静态逻辑架构 -> 动态时序处理 -> 状态变更反馈”的完整路径。
      - **细节注入：** 在描述流程时，强制引入具体的参数类型（如：16进制指令集、JSON 报文结构、毫秒级超时时间等）和逻辑判断准则（If-Then-Else 逻辑分支）。
      - **衔接词增强：** 除了因果词，必须加入空间/时间维度的定位词，如“在逻辑层的入口处”、“在完成数据清洗后的 T+1 阶段”、“位于存储引擎与查询接口的中间层”等，以增加空间感和可信度。
     - 模块作用与输入/输出：至少覆盖 3 个模块；每个模块至少 1 段连续文字，并在该段中自然包含输入、处理逻辑、输出（不要求每段固定条数，但必须可落地、可追溯）。在描述模块处理逻辑时，不仅要写本模块做了什么，还必须交代：1. 它如何与其他模块协同工作；2. 它如何处理异常情况；3. 该逻辑如何直接支撑了后文提到的技术效果。
     - 连接关系（数据流/控制流/触发条件）：将关键链路写成可落地的“连续链路描述”，至少 3 条链路；每条链路至少交代清楚“从哪里到哪里 + 为什么这样触发/调用”。
     - 关键实现步骤（编号步骤）：至少 6 步；以编号步骤叙述为主，每步用 1-2 句描述输入/输出或前后置条件，确保读者能按步骤复现实现路径。
     - 关键改进点如何实现目的（实现-效果映射）：至少 3 组；每组用连续文字写“实现了什么（做了什么）—解决了哪个技术问题/缺陷—带来了什么技术效果”，并在必要处引用改进点编号与第 1 部分缺陷编号。“每组‘实现-效果映射’逻辑上必须包含：现有技术手段描述、本申请的技术干预动作、底层逻辑状态的改变、最终的技术性能提升。”
     - 替代方案：至少 2 个“可选实施方式”，用连续文字说明与主实施方式的差异点（至少指出 2 处差异），并简单说明各自适用条件或取舍。
     - 图示插入策略（独立 PNG 优先，失败回退无 Mermaid 代码）：优先基于“图示逻辑（节点/连线语义）+ 图示解释”调用图像生成能力生成独立 PNG。成功时在最终 Markdown 中插入图片引用 `![图X](...)`；失败时在该图后紧跟一句简短注释：`图X PNG 生成失败原因：<失败原因>`，并额外给出“图逻辑文本表示”（至少包含节点列表与关键连线列表）用于可追溯。
     - 强制约束：最终保存的 Markdown 中。
     - 并且在每张图的图片引用（或失败原因注释与图逻辑文本表示）后，紧跟“图示解释”（图注要点 + 逐节点解释 + 逐连线解释），以保证与第 3 部分连接关系/实现步骤一致
5. `## 4．申请本发明的目的`
   - 默认写“保护研发成果为主”的通用口径（若用户明确给出评职称/高新/项目，则替换成对应表述）
6. 文末“待补字段清单”（只列缺失项）
7. 保存到文件（必须）
   - 默认保存目录：`./patent_outputs/`
   - 默认文件名规则：`{发明设计名称(去除非法字符)}-{时间戳}.md`（时间戳精确到分钟即可）
   - 若用户在对话中明确给出保存路径/文件名，则使用用户提供的路径
   - 保存成功后，在聊天里回显一行：`已生成文件：<保存路径>`
   - 如果当前运行环境不支持落盘：则回退为仅输出聊天文本，并在末尾提示：`（无法写入文件：原因未知）`

### 5) 生成后的质量闭环（自检，不要向用户输出超长检查报告）
你在输出前自检：
1. 第 3 部分是否写到“具体改进了哪些地方以及如何改进”而非仅原理/仅功能介绍
2. 是否有清晰的连接关系/数据流/控制流，并能从图示解释追到实现步骤
3. 替代方案是否至少 1 条
4. 第 1 部分的缺陷是否能在第 3 部分找到对应解决点
5. 第 3 部分是否足够丰满：关键实现步骤至少 6 步、覆盖至少 3 个模块（用连续段落表达输入/处理/输出）；且每张图都有“图注 + 逐节点/逐连线解释”

## Output Format（固定）
你必须使用以下 Markdown 标题与大纲层级输出最终结果：
- `# 专利技术交底书（发明专利，Markdown版）`
- `## 1．现有技术的现状及缺陷`
- `## 2．本发明的整体构思`
- `## 3．本发明技术方案的详细阐述`
- `## 4．申请本发明的目的`

并在第 3 部分按需插入：
- 图 1/图 2 的图片引用（PNG），并在每张图后附“图示解释”

## Examples
示例输入（用户）：
- “我想发明一个用于数据治理的字段敏感度自动排序系统，能降低 LLM 幻觉导致的排序冲突。现有方案是逐字段分类再汇总，准确率不稳。”

示例输出（骨架/确认阶段，摘要示意）：
- 先回显技术对象/模块草案，并询问：现有方案模块是什么、你希望改进的关键步骤是什么、最终系统里数据流从哪里到哪里。
