# 阶段 1：领域问题与难点（flowtest）

**课题聚焦**：在 GDPR / PIPL 等合规背景下，对**大规模（约 10^5 级）数据库字段**做敏感度分级（如 L1–L5）。目标是在可接受成本下得到**可审计、全局自洽**的排序或等级，而非孤立噪声标签。

---

## 1. 领域难点在哪？

1. **尺度不绝对**：L1–L5 是政策与业务语境下的离散档，缺乏像「米/千克」那样的客观量纲；模型需在**隐含量尺**上对齐十万条异构字段（名、注释、取值模式）。
2. **边界语义模糊**：准标识符、间接敏感、组合重识别等场景，**单字段绝对判断**不稳定，易产生合规争议。
3. **规模与成本**：若用成对比较穷举，比较次数随 \(N\) 二次增长；真实 LLM 调用预算有限，必须在**采样/传播**与**精度**间折中。
4. **全局一致性**：局部判断聚合后可能出现**循环偏好**或与传递性不一致，难以直接导出全序或稳定等级映射。
5. **可解释与审计**：企业内审常要求说明「为何该字段高于彼字段」；纯黑盒分类分数往往不足。

---

## 2. 目前前沿 / 既有方案解决了哪些难点？

| 路线 | 大致做法 | 已缓解的难点 |
|------|----------|----------------|
| **规则与词典** | 关键字、正则、数据分类标准映射 | 常见 PII 的**高召回**、部署简单 |
| **Pointwise 分类** | 每字段独立预测等级（传统 ML 或 LLM） | **吞吐高**、工程上易并行；对「明显高敏/明显公开」字段往往够用 |
| **Pairwise / 列表排序（IR）** | RankNet、LambdaRank、LLM 重排（如 RankGPT） | **相对比较**比绝对分更稳；有成熟理论与实现 |
| **成对比较聚合** | Bradley–Terry、谱方法、Rank Centrality | 从噪声成对观测恢复**全局得分/排序** |
| **Embedding 检索** | 语义近邻、聚类 | 把「该和谁比」限制在**局部邻域**，降低无效比较 |

---

## 3. 遗留难点与原因

1. **Pointwise 在边界与幻觉上仍弱**：单次调用缺锚点，易「同类不同级」；与 §1 中尺度、边界问题直接相关。
2. **全 Pairwise 不可扩展**：\(O(N^2)\) 与 LLM 成本在 \(N \sim 10^5\) 下不可接受；**如何用结构化先验（如 Top-K 相似）压缩比较次数**仍依赖具体数据与超参，需实证。
3. **LLM 成对判断仍有错**：聚合后不一致；**何种全局修正（如基于主特征向量的迭代）在噪声下更稳**需与基线对照验证。
4. **排序到 L1–L5 的映射**：全序得分如何映射到法规档位、阈值如何校准，属于**业务与评测设计**，技术 pipeline 需显式留出环节。
5. **可复现评测**：真实 schema 难公开；**合成与脱敏子集上的协议**（划分、金标准、基线）必须写清，否则「提升 X%」不可审计。
